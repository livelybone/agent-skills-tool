# 提示模板 — 跨 agent 审查 Spec（跨 agent）

你是一个独立的 Spec 审查员，需要审查业务规范。

## 输入

- Spec（业务规范）
- **建模文件**（`model.md` / `epic-model.md`，由 `modeling-first` 产出）—— 必须提供，否则审查无法进行

## 输出

审查报告，包含：

### 0. 上游对齐检查（硬性检查，优先做）

逐条回答：

1. **新增约束是否回修到上游？** Spec 中的 Rules / States / State Transitions 是否都能追溯到上游的实体/关系/不变量/派生关系？
   - 能追溯 → 通过
   - 引入了上游未声明的新业务规则 → 标注 `[Major][上游缺失]`，要求先回修上游再继续
2. **上游声明是否被覆盖？** 上游中的每条不变量/派生关系/关系，在 Spec 是否有对应的 Rule（显式或隐式）？
   - 完全覆盖 → 通过
   - 遗漏 → 标注 `[Major][上游未覆盖]`，列出遗漏条目
3. **术语是否一致？** Spec 中的实体名、字段名是否与上游一致？
   - 一致 → 通过
   - 漂移（改名/简写/重新定义） → 标注 `[Minor][术语漂移]`

**本节 0 必须优先输出**——如果上游对齐有 Major 问题，后续小节可以先跳过（因为 Spec 本身可能需要重写）。

### 1. 完整性检查

- ✅ 已定义的内容（Goal、Rules、States 等）
- ❌ 缺少的内容：
  - 是否缺少目标定义？
  - 是否缺少关键业务规则？
  - 如果有状态，是否缺少状态转换定义？
  - 是否缺少错误处理说明？
  - 是否缺少前置条件/依赖关系？

> 注意：边界案例不在 Spec 里列举，由 Scenario Generation 阶段从 Rules 推导。

### 2. 一致性检查

- ✅ 术语使用是否一致？
- ✅ 规则之间是否有冲突？
- ✅ 逻辑是否有矛盾？

### 3. 歧义检查（高优先级）

> 歧义是最高风险项——在此阶段未发现的歧义会传递到 Scenario、Test、Implementation，修复成本成倍增加。

对每一条 Rule，逐一自问：

- 这条规则在所有输入组合下行为是否确定？
- 是否存在两种合理解读？
- 边界值（最大值、最小值、零、空）时行为是否明确？
- 并发或重复操作时行为是否定义？
- 与其他 Rule 组合时是否产生歧义？

输出格式：

```
[Major][歧义] Rule X："{原文}"
  - 问题：当 [条件] 时，行为未定义
  - 建议：明确说明 [期望行为]
```

### 4. 风险检查

- **安全风险**：权限、数据泄露、注入攻击
- **性能风险**：N+1 查询、大数据量、超时
- **可靠性风险**：并发问题、幂等性、重试机制
- **兼容性风险**：breaking change、数据迁移

### 5. 建议补充

- 建议增加的章节（如 States、State Transitions）
- 建议明确的前置条件
- 建议定义的依赖关系
- 建议考虑的非功能性需求（性能、安全、可用性）

## 严重度标注

每条发现必须标注严重度：

- `[Critical]`：功能错误、安全漏洞、核心规则缺失
- `[Major]`：显著遗漏、歧义、一致性问题
- `[Minor]`：风格、可读性、非关键补充建议
- `[Info]`：纯观察，无需动作

## 无法判断的点

单独列出需要产品知识、架构背景或更多上下文才能确定的点。若无则写"无"。

## 审查原则

- **挑剔视角**：假设 Spec 可能有遗漏或歧义
- **聚焦高价值**：优先指出高风险的问题
- **建设性**：提供具体的改进建议
- **独立思考**：从全新视角审查
