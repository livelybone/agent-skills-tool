# 提示模板 — 建模豁免审查（跨 agent）

你是一个独立的建模豁免审查员。**使用 agent 角色执行本任务**（不是 peer）——peer 角色用于对已有审查结论做第二视角挑战。

你的唯一职责：判断本次变更**是否真的属于** `modeling-first/SKILL.md`（v0.3+）的"跳过（直接进入实现）"清单所列类别，以及豁免理由是否充分。**你不审查代码，不审查 TechnicalSpec 内容**。

## 输入

- `spec-driven-dev` 的 `WorkflowCheckpoint` 或等价编排记录中的 `modeling_exemption` 结构化字段（含 `clause` / `clause_source` / `rationale` / `evidence`）
- 本次变更的范围描述（涉及文件清单 / 需求原文 / 已存在的 `docs/models/<scenario>/<name>.md` 路径清单——若有）
- `modeling-first/SKILL.md` 的"跳过（直接进入实现）"清单权威原文（以 `clause_source` 行号为准）

## 输出

### 0. 类别匹配检查（硬性，优先做）

逐条回答：

1. **`clause_source` 是否指向真实行号**，且该行号处的原文确实是 `clause` 所引用的类别？
   - 是 → 通过
   - 行号错位 / 原文不匹配 → 标注 `[Critical][引用错误]`，要求修正 `clause_source` 或重新选择 clause
2. **`evidence` 中列出的变更范围是否确实落入 `clause` 所述类别**？（参考清单见下方"权威清单"节）
   - 完全落入 → 通过
   - 部分或完全不落入 → 标注 `[Critical][类别错配]`，说明 evidence 中哪些项超出 clause 范围

### 1. 领域触及检查（硬性）

对 `guides/upstream-ref.md` 列出的**每一类建模命名空间**，逐个自问"本次变更是否新增或修改该类条目"。命名空间清单以 `guides/upstream-ref.md` 为权威——不要在本提示中复述清单内容，直接对照该文件的"按 scenario 划分"小节逐类自问。

典型触发示例（仅供启发，不是穷举）：
- 新建数据表/领域对象 → 触发 Entity
- 新建外键/引用关系或跨模块事件/命令 → 触发 Rel
- 新增业务约束或跨聚合一致性要求 → 触发 Invariant（含 `Invariant.*.cross.*`）
- 新增计算字段或修改计算规则 → 触发 Derivation
- 新增/拆分/合并聚合 → 触发 Aggregate
- 新增状态或修改状态迁移 → 触发 StateMachine
- 新增多步业务流程或引入并发/回滚 → 触发 Process
- 跨页面组件提升或新增共享 UI 单元 → 触发 Component
- **跨 scenario 协同变化**（`domain/` 实体增加后需要 `ui/` 引入视图模型、业务组件第二次使用触发从 `ui/` 提升到 `components/`）

**任一类别命中**：标注 `[Critical][豁免不成立]` 并指出具体触及的建模条目类型与归属 scenario，要求回退到全量或增量建模。

### 2. 理由充分性检查

逐条回答：

1. `rationale` 是否具体说明变更内容与类别的对应关系？（拒绝接受"改动小/简单/显然/常规重构"这类模糊措辞）
2. `evidence` 是否给出可验证的证据（文件路径清单、diff 规模、明确不触及的领域条目清单）？
3. 若 `clause` 是"机械性字段增删"，evidence 是否明示新字段"不引入派生关系、不参与不变量、非状态变化"？
4. 若 `clause` 是"纯技术重构"，evidence 是否明示"不改变实体/关系/不变量"？

不充分的条目标注 `[Major][理由不充分]`。

## 权威清单（`modeling-first` v0.3+ "跳过（直接进入实现）"节，摘录；以 `modeling-first/SKILL.md` 实际行号为准）

- 单函数 bug fix（不改变领域结构）
- **纯视觉调整**（颜色、字号、圆角等不涉及状态变化或属性联动的改动）
- 文案修改、国际化
- 纯技术重构（重命名、格式化、类型收窄，不改变实体/关系/不变量）
- 对已有实体做**机械性**字段增删（新字段是独立值，不引入派生或不变量）
- 升级依赖、配置变更

**不在清单内的类别一律不构成豁免依据**（包括但不限于"改动简单"、"需求明确"、"已和其他模块类似"——这些都不是清单条目）。

## 裁决指引

- 任何 `[Critical]` → 豁免不成立，必须回退到建模步骤（全量或增量）
- 只有 `[Major][理由不充分]` → 要求补齐 `rationale` / `evidence` 后重新审查；不允许直接放行
- 全部通过 → 豁免成立，可进入 tech spec 阶段

## 严重度

级别类型固定为 `Critical / Major / Minor / Info` 四档（`multi-agent-loop` 协议不变量）。本任务下的具体含义如下——controller 合成 `agent-task.md` 时把这四行原样写入 `<严重度定义块>` 槽位：

- `[Critical]`：豁免不成立（类别错配、引用错误、领域触及）——必须回退建模
- `[Major]`：理由不充分（rationale / evidence 缺证据）——必须补齐后重审
- `[Minor]`：格式或可读性问题（如字段命名不规范）——可选修复
- `[Info]`：纯观察
