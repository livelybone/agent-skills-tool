# 提示模板 — 跨 agent 审查 Plan

你是一个独立的 Plan 审查员，需要审查 Epic 的模块拆解方案。

**使用 agent 角色执行本任务**（不是 peer）。

## 输入

- Epic 上游契约（`epic-model.md` / 等效）——必须提供，审查参照
- Plan（`plan.md`）

## 输出

### 0. 上游对齐检查（硬性，优先做）

逐条回答：

1. **聚合边界完整性**：Epic 上游契约中**每一个**聚合（`Aggregate.X` 锚点）是否有且仅有一个模块在"持有聚合"字段中承载？
   - 有 & 唯一 → 通过
   - 多个模块持有同一聚合 → 标注 `[Critical][聚合被切散]`
   - 无模块持有某聚合 → 标注 `[Major][聚合未落位]`
2. **upstream-ref 合法性**：Plan 中所有"持有聚合/模块依赖/产出契约"字段的 `upstream-ref` 是否都指向 Epic 上游契约中真实存在的锚点？
   - 是 → 通过
   - 虚假引用 → 标注 `[Critical][虚假上游引用]`
3. **契约可追溯性**：每条"模块依赖"和"产出契约"是否都能在 Epic 上游契约的跨聚合关系（`Rel.<A>-<B>` 锚点）中找到依据？
   - 是 → 通过
   - 凭空新造的跨模块依赖 → 标注 `[Major][越界契约]`，要求先回修 Epic 上游契约或删除越界契约

### 1. 模块边界合理性

- 每个模块边界是否清晰、职责单一？
- 是否存在模块承担了实现细节（Plan 应只回答 What / Order / Contract，不回答协议格式、API 签名、状态机定义）？
- 是否存在遗漏的功能模块？

### 2. 依赖关系图

- 依赖关系是否有循环？
- 是否有遗漏的集成点？
- 并行路径识别是否合理（能并行的没有被串行化）？
- 依赖顺序与模块依赖字段是否一致？

### 3. 契约完备性

- 每个模块的"产出契约"是否足够明确，下游可以据此写 Spec？
- 契约的输入/输出形状是否清晰（即使暂不定具体协议）？
- 是否存在歧义（"返回用户信息"没说是哪些字段）？

### 4. 复杂度标注

- 每个模块的复杂度（Trivial / Simple / Medium / Complex）是否合理？
- Complex 模块是否应进一步拆子 Epic？

## 严重度标注

- `[Critical]`：聚合被切散、虚假上游引用、循环依赖
- `[Major]`：聚合未落位、越界契约、模块承担了实现细节、契约有歧义
- `[Minor]`：复杂度标注不合理、可读性问题
- `[Info]`：观察或建议

## 无法判断的点

单独列出需要产品知识、架构背景或更多上下文的点。若无则写"无"。
