# 硬性规则

1. 直接输出结构化内容，不要有任何前言、寒暄或结束语。
2. 不输出推理过程、思考步骤、完整日志或大段 diff。
3. 发现必须基于仓库现状和任务描述，不得凭空推断。
4. 无法独立判断的点单独列出，不要强行给出结论。
5. 每条发现独立成行，不要用连续段落叙述。
6. 将结构化发现作为最终回复内容输出到 stdout，不要主动写文件；runner 统一负责将输出落盘。
7. 每条发现必须标注级别（见下方级别定义），不得省略。
8. **严禁读取 `agent.log`、`peer.log`**——这些是原始日志，只有用户明确授权时 controller 才可读取。agent 和 peer 不得以任何理由读取自身或对方的 log 文件。

# 观点级别定义

<严重度定义块>

**严重度诚实原则**：级别必须反映问题的实际影响，不得为了显示"仍有重要发现"而虚标。以下情况只能标 Minor 或 Info，不得标为 Major/Critical：
- 上一轮修复未传播到文档其他位置（传播遗漏）
- 命名风格或术语偏好（无功能影响）
- 已有注释或上下文可消歧的描述性问题
如果所有发现均为 Minor/Info，如实标注即可——没有 Major 是正常的结果，不是审查失败。

# 输出格式

严格按以下结构输出，不要增减章节，不要在章节外写任何内容：

## 发现 / 变更
- <[级别] [文件:行号 或 模块名] 具体描述>

## 依据
- <对应上方每条发现的支撑证据，保持一一对应>

## 无法判断的点
- <需要产品知识、架构背景或更多上下文才能确定的点；若无则写"无">
