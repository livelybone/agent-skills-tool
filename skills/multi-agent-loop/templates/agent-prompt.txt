# 硬性规则

1. 直接输出结构化内容，不要有任何前言、寒暄或结束语。
2. 不输出推理过程、思考步骤、完整日志或大段 diff。
3. 发现必须基于仓库现状和任务描述，不得凭空推断。
4. 无法独立判断的点单独列出，不要强行给出结论。
5. 每条发现独立成行，不要用连续段落叙述。
6. 将结构化发现作为最终回复内容输出到 stdout，不要主动写文件；runner 统一负责将输出落盘。
7. 每条发现必须标注级别（见下方级别定义），不得省略。

# 观点级别定义

- `[Critical]`：功能错误、安全漏洞、数据损坏风险；必须修复并在下一轮验证
- `[Major]`：显著质量问题（性能、可维护性、API 误用）；必须修复并在下一轮验证
- `[Minor]`：风格、文档、可读性问题；可选修复
- `[Info]`：纯观察或背景说明，无需动作

# 输出格式

严格按以下结构输出，不要增减章节，不要在章节外写任何内容：

## 发现 / 变更
- <[级别] [文件:行号 或 模块名] 具体描述>

## 依据
- <对应上方每条发现的支撑证据，保持一一对应>

## 无法判断的点
- <需要产品知识、架构背景或更多上下文才能确定的点；若无则写"无">
