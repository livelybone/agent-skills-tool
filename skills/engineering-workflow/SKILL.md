# 基础工程流程

## 适用场景
- 新功能开发与实现
- 新项目初始化与架构搭建
- 现有功能的重构或优化
- 需要系统性规划的编码任务

## 必须材料
- 需求描述与验收标准
- 约束条件（时间/依赖/环境）
- 风险点与优先级

## 执行步骤
1. 明确目标、范围与约束，补齐最小输入集
2. 调研复用方案并做取舍记录
3. 以 MVF（最小可用功能单元）为单位拆分任务并规划交付顺序
4. 实施开发并遵循质量与一致性规范
5. 运行验证（类型检查/单测/验收）并记录结果
6. 总结产物与决策，准备交付

## 产物与格式
- **Plan**：阶段/任务清单，按依赖顺序排列
- **Decision Log**：关键取舍与理由，包含复用方案评估
- **Verification**：验证步骤与结果，包含测试/类型检查输出

## 质量门槛

> 遵循全局上下文中的"代码质量基础规范"
> 遵循全局上下文中的"复用优先流程"
> 遵循全局上下文中的"架构一致性原则"

### 核心原则
- **MVF 优先**：最小可用功能单元优先，按依赖顺序交付可运行成果
- **测试优先**：先明确用例或测试计划，不适用需给替代验证路径

### 检查清单（核心）

**新功能开发前**：
- [ ] 已搜索成熟方案（至少 3 个）
- [ ] 已检查项目内现有功能
- [ ] 已确定复用/实现方案
- [ ] 已确认输入信息最小集（需求/验收/约束/依赖）
- [ ] 已确定单元测试用例或测试计划

**代码编写时**：
- [ ] 遵循全局上下文的长度限制
- [ ] 命名具有描述性
- [ ] 无重复代码模式

**任务完成时**：
- [ ] 每个功能单元完整可运行
- [ ] 代码风格与项目一致
- [ ] 完成必要验证（类型检查/单元测试等）

详细检查清单见 `references/checklists.md`

## 验证方式

> 遵循全局上下文中的"验证方式通用流程"

### 本 skill 特定的验证
- **功能验证**：每个 MVF 可独立运行并通过验收标准
- **类型检查**：如项目配置了类型系统，运行类型检查命令
- **单元测试**：运行测试命令并确保通过
- **集成验证**：验证与现有模块的集成无冲突

## 不覆盖范围
- 纯讨论、方案评估、策略建议（不涉及实际编码）
- 概念解释、技术咨询（不需要工程动作）
- 具体的测试编写（由 unit-test skill 负责）
- 具体的代码评审（由 code-review skill 负责）
