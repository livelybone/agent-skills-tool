# 代码评审专项

## 适用场景
- PR 提交前的质量检查
- 代码合并前的安全/性能审计
- 上线前最终评审
- 团队成员代码互审

## 必须材料
- PR 链接或 diff
- 关联需求/问题描述
- 风险等级与重点关注维度
- 已有测试结果与验证步骤

## 执行步骤
1. 声明已加载 code-review SKILL 并遵循其规则
2. 明确评审范围与优先级（质量/安全/性能/架构）
3. 选择适用的检查清单并说明理由
4. 逐项检查并记录问题（含可验证路径）
5. 输出评审结果与优先级
6. 修复后进行验证性复查（如适用）

## 产物与格式

**评审结果文档**：

```markdown
# 代码评审结果

## 评审范围
- 文件：xxx.ts, yyy.py
- 重点维度：代码质量、安全

## 问题列表

### P0 (Critical)
- [ ] [文件名:行号] 问题描述
  - 修复建议：具体的修复方法
  - 验证方式：如何验证修复有效

### P1 (High)
...

## 评审总结
- 通过项：X 项
- 待修复项：Y 项
- 建议优先修复：P0、P1 级别问题
```

**严重程度分级**：

| 级别 | 定义 | 示例 |
|------|------|------|
| P0 (Critical) | 安全漏洞、数据破坏风险 | SQL 注入、敏感数据泄露 |
| P1 (High) | 功能缺陷、性能严重问题 | 内存泄漏、死循环 |
| P2 (Medium) | 代码质量问题、小性能问题 | 重复代码、N+1 查询 |
| P3 (Low) | 可读性、风格不一致 | 命名不规范、注释缺失 |

## 质量门槛

> 遵循全局上下文中的"代码质量基础规范"

### 核心检查维度
- **质量**：是否符合代码质量基础规范（长度、命名、可读性）
- **安全**：是否存在安全漏洞（详见 references/security-checklist.md）
- **性能**：是否存在性能问题（详见 references/performance-checklist.md）
- **架构**：是否符合架构一致性原则

### 检查清单（核心）

**强制检查**（遵循全局上下文）：
- [ ] 无硬编码密码、token、API 密钥
- [ ] 无敏感数据泄露
- [ ] 无 SQL 注入、XSS 等安全漏洞

**代码质量检查**：
- [ ] 遵循长度限制（文件 ≤ 500 行，函数 ≤ 50 行）
- [ ] 命名规范（描述性、一致性）
- [ ] 无重复代码模式
- [ ] 无硬编码配置

**架构检查**：
- [ ] 遵循现有目录结构
- [ ] 遵循项目命名约定
- [ ] 无循环依赖

详细检查清单见：
- `references/quality-checklist.md`（质量检查）
- `references/security-checklist.md`（安全检查）
- `references/performance-checklist.md`（性能检查）

## 验证方式

> 遵循全局上下文中的"验证方式通用流程"

### 本 skill 特定的验证
- **自动化检查**：运行 lint、类型检查、安全扫描工具（如可用）
- **手动审查**：逐文件检查 diff，标注问题行号
- **修复验证**：修复后重新运行检查，确认问题已解决

## 不覆盖范围
- 简单语法错误修复（可由通用知识完成）
- 仅限格式化/风格调整的任务
- 性能 profiling（需专门的性能测试）
