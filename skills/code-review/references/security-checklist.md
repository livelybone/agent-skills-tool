# 安全漏洞检查清单

## 注入攻击

```
□ 所有用户输入都经过验证和转义
□ SQL 查询使用参数化查询（prepared statement），不拼接字符串
□ 命令执行避免直接使用用户输入
□ XSS 防护：输出到 HTML 的数据经过转义
□ 路径遍历：文件路径不直接使用用户输入
□ LDAP/XML/模板注入：相关输入经过验证
```

## 权限控制

```
□ 所有需要认证的接口都有认证检查
□ 所有需要授权的操作都有权限检查
□ 权限检查在业务逻辑执行前进行
□ 避免客户端权限判断（必须在服务端验证）
□ 敏感操作有二次验证（如删除、支付）
□ 会话管理：token 有过期时间，敏感操作后刷新 token
```

## 数据泄露

```
□ 敏感信息（密码、密钥、token）不记录到日志
□ 敏感信息不存储在客户端（localStorage、cookie）
□ 密码使用安全的哈希算法（bcrypt、argon2），不使用 MD5/SHA1
□ API 响应不包含不必要的敏感信息
□ 错误信息不泄露系统内部信息（堆栈、路径）
□ 敏感数据传输使用加密（HTTPS）
```

## 依赖安全

```
□ 第三方依赖无已知高危漏洞
□ 依赖版本不过期（1年以上未更新需评估）
□ 不使用已废弃的库
□ 依赖的完整性有验证（checksum、签名）
```

## 加密与哈希

```
□ 不使用弱加密算法（DES、RC4）
□ 不使用弱哈希算法（MD5、SHA1）
□ 加密密钥不硬编码在代码中
□ 随机数生成使用密码学安全的随机数生成器
□ 敏感数据存储时加密
```

## CSRF 防护

```
□ 状态改变的操作（POST/PUT/DELETE）有 CSRF 防护
□ 使用 CSRF token 或 SameSite cookie
□ 避免在 GET 请求中执行状态改变操作
```

## SSRF 防护

```
□ 用户输入的 URL 有白名单验证
□ 避免直接使用用户输入发起网络请求
□ 内部服务地址不对外暴露
```

## 文件上传

```
□ 文件类型有严格验证（不仅检查扩展名，还检查文件头）
□ 文件大小有限制
□ 上传的文件不在 Web 根目录或执行目录
□ 文件名经过清理，避免路径遍历
□ 图片上传有尺寸限制，防止 DoS
```

## 业务逻辑安全

```
□ 金额计算使用 Decimal，不使用 float
□ 并发操作有锁或版本控制，防止竞态条件
□ 重要操作有幂等性保证
□ 数量、金额等有合理上下限检查
□ 业务流程不可跳过关键步骤
```

## API 安全

```
□ API 有频率限制（rate limiting）
□ 敏感 API 有额外的认证（如验证码）
□ API 响应时间不因数据存在性不同而泄露信息
□ 批量操作有数量限制
□ API 版本管理清晰
```

## 配置安全

```
□ 生产环境关闭 debug 模式
□ 敏感配置从环境变量或密钥管理服务读取
□ 默认账号密码已修改
□ 不使用的功能、端口已关闭
```
