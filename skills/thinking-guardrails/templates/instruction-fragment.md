## 编码前置思维守卫

> 以下两条原则适用于所有涉及代码编写的任务，写代码前必须过检查。
> 完整背景与反模式案例见 thinking-guardrails skill 的 SKILL.md

### 建模思维

写代码前先建模（识别实体、关系、派生链、不变量），不要模式匹配（见到需求直接套模板）。

**前置检查**（涉及新领域信息时必须回答；简单改动心里过一遍即可）：

1. 涉及哪些领域实体？（去掉技术词，保留领域词）
2. 实体之间什么关系？（基数、所有权、删除语义）
3. 哪些值是派生的？（能从别的值算出来的不暴露为独立输入）
4. 什么状态是非法的？（不变量 → API 设计 + 测试用例）
5. 项目里已有表达吗？（复用/扩展优先于新建）

**反模式信号**：函数 5+ 参数且多个可派生；项目已有功能相同的 utility 却新建了一个；同一实体行为散落在 Service/Manager/Handler 多个文件；数据结构模仿 API 响应而非领域概念。

### 奥卡姆剃刀

每增加一个东西（prop、参数、配置、wrapper、抽象层、类型、文件），都要问"去掉它会 break 什么"。说不出来就不要加。

**前置检查**（增加任何东西前必须回答）：

1. 这个问题属于哪一层？（运行时 / 类型系统 / 测试分层 / 构建配置 / 文件命名约定——若新增物存在理由是"补齐另一层的漏洞"，先回那一层修根因，别在这层加补丁）
2. 去掉它会 break 什么？（说不出 → 不加）
3. 解决的是真实问题还是假想问题？（"万一将来需要"不是理由）
4. 能用已有的东西表达吗？（新建前先搜索）
5. 根变量是什么？（有派生关系的只暴露根变量）

**反模式信号**：新增参数但调用方总传同一值；wrapper 只有一个调用方；配置项只有一个合法值；类型定义只在一处使用；文件只导出一个 < 20 行的函数；**新增物是错层补丁**（runtime wrapper 为兼容测试切 `Platform.OS`、`.d.ts` 补 tsc 不认的平台后缀、`types.ts` 仅统一同目录 import）→ 回问题所属的层级（测试分层/文件命名约定/内联类型）。
