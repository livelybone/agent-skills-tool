# Thinking Guardrails

> **定位**：原则包（Principle Package），不是工作流 skill。定义两条通用思维原则及其检查清单，供其他 skill 引用或注入项目级 CLAUDE.md 始终生效。

## 问题

LLM 编码的两个系统性缺陷：

1. **模式匹配替代建模**：见到需求 → 检索训练数据中相似代码模板 → 产出。结果是同一概念多处重复实现、应派生的值被当独立输入、与现有抽象平行新建相似概念。
2. **实体膨胀**：默认倾向"加东西"（多加个参数更安全、多一层封装更灵活、多一个 flag 更通用），不会主动问"去掉它会 break 什么"。

这两个问题跨技术栈、跨项目普遍存在，不应绑定到特定工作流。

## 两条原则

### 原则 1: 建模思维（Modeling Thinking）

**写代码前先建模，不要模式匹配。**

建模 = 识别领域中的实体、关系、派生链、不变量，从模型派生实现。
模式匹配 = 见到需求直接套模板产出代码。

#### 前置检查（写代码前必须回答）

1. **这段代码涉及哪些领域实体？** 列出名词，去掉技术词（"组件"、"接口"），保留领域词（"订单"、"权限"）
2. **实体之间什么关系？** 基数（1:1/1:N/N:N）、所有权（谁持有谁）、删除语义（级联/保留/禁止）
3. **哪些值是派生的？** 对每个字段问"能从别的值算出来吗？"——能算的不暴露为独立输入，内部计算
4. **什么状态是非法的？** 不变量决定 API 设计（派生值不暴露 setter）和测试用例（每条不变量至少一条测试）
5. **项目里已有表达吗？** 搜索现有代码，复用/扩展优先于新建

> 不是每次写代码都要产出 `model.md`——简单改动（bug fix、样式、配置）心里过一遍即可。判断标准：**需求是否引入新的领域信息**（新实体/新关系/新不变量/新派生关系），不是改动大小。

#### 反模式信号

- 一个函数接收 5+ 个参数，其中多个可以从其他参数算出来 → 派生值被当独立输入
- 新建了一个 utility/helper，但项目里已有功能相同的 → 平行新建
- 同一实体的行为散落在 Service/Manager/Handler/Controller/Helper 多个文件 → 无模型，按技术职责切空壳
- 数据结构的字段名和形状模仿了 API 响应而非领域概念 → 数据结构模仿实现

### 原则 2: 奥卡姆剃刀（Occam's Razor）

**若无必要勿增实体。每增加一个东西，都要问"去掉它会 break 什么"。说不出来就不要加。**

"实体"在这里是广义的：prop、参数、配置字段、wrapper 层、抽象层、中间变量、类型定义、文件、目录——任何增加系统复杂度的东西。

#### 前置检查（增加任何东西前必须回答）

1. **去掉它会 break 什么？** 说不出具体的 break 场景 → 不加
2. **它解决的是真实问题还是假想问题？** "万一将来需要" 不是理由；"现在有 3 处代码因为缺它而重复" 是理由
3. **能用已有的东西表达吗？** 新建前先搜索；能扩展已有抽象就不新建
4. **根变量是什么？** 多个值之间有派生关系的，只暴露根变量，其余内部计算

#### 反模式信号

- 新增了一个 prop/参数，但调用方总是传同一个值 → 不需要这个参数
- 新增了一层 wrapper/adapter，但只有一个调用方且没有替换需求 → 过早抽象
- 新增了一个配置项，但只有一个合法值 → 硬编码即可
- 新增了一个类型定义，但只在一处使用且内联后可读性不变 → 不需要
- 新增了一个文件，但只导出一个 < 20 行的函数 → 合并到调用方

## 执行步骤

### Setup（首次触发时执行，必须完成）

本步骤是必须的。将 `templates/instruction-fragment.md` 的内容写入当前平台实际加载的全局指令文件。未完成此步骤，skill 不视为已安装。

#### 1. 确定当前平台的指令文件

根据当前运行的 agent 平台，定位实际加载的全局指令文件：

- **Claude Code**：全局 `CLAUDE.md`（或工作区级，仅当它是宿主实际入口时）
- **Codex / OpenCode**：全局 `AGENTS.md`（运行时实际读取的那个）
- **Gemini**：全局 `GEMINI.md`（或运行时系统指令文件）

不要写入未被宿主加载的项目本地文件——那不算完成 setup。

#### 2. 注入片段（非破坏性）

读取 `templates/instruction-fragment.md`，将其内容追加到上述指令文件中。

注入规则：

- 检查指令文件是否已包含 `## 编码前置思维守卫` 段。已存在则跳过，不重复注入
- 不替换指令文件中已有的内容，仅追加
- 如果指令文件中已有类似原则（如"若无必要勿增实体"），片段作为**补充**追加（增量价值在于结构化检查清单和反模式信号列表），不删除已有内容

#### 3. 验证注入

确认指令文件中包含 `## 编码前置思维守卫` 段。若缺失，setup 未完成。

#### 4. 多平台项目

若项目同时使用多个 agent 平台，对每个平台重复步骤 1-3。片段内容相同，注入目标不同。

### 日常：无显式步骤

Setup 完成后，原则通过指令文件始终在上下文中。coding agent 在写代码前自行过前置检查，不需要手动触发 skill。

## 产物与格式

### templates/instruction-fragment.md

本 SKILL.md 是原则的真理源（含完整背景、案例、反模式解释）。片段是其**派生物**——精简为检查清单和反模式信号，去掉背景说明，可独立理解。修改原则时先改 SKILL.md，再同步片段。一份片段，多平台复用。

> 片段相对于用户指令文件中可能已有的通用设计原则（如"若无必要勿增实体"），增量价值在于：**结构化的前置检查清单**（写代码前必须回答的具体问题）和**反模式信号列表**（可对照的具体 code smell）。

## 验证方式

> 遵循全局上下文中的"验证方式通用流程"

### 本 skill 特定验证

1. 确认当前平台的全局指令文件包含 `## 编码前置思维守卫` 段。缺失则 setup 未完成
2. 打开一个新会话，不手动触发任何 skill，给一个涉及新实体的编码需求——agent 是否自动执行了前置检查？
3. 给一个倾向于过度设计的需求（如"做一个通用的 XX 框架"）——agent 是否用奥卡姆剃刀挑战了不必要的复杂度？
