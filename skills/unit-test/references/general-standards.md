
# 通用单元测试标准

## 结构与命名

- 结构：Arrange-Act-Assert 或 Given-When-Then
- 测试粒度：单个行为/边界一条用例
- 命名规范：优先遵循项目既有规范；无既有规范时使用 `should <expected> when <condition>`
- 描述必须表达“预期 + 条件”，避免模糊词

示例：

- `should return default value when input is empty`
- `should throw when user is not authorized`

## 隔离与确定性

- 不访问真实网络、数据库、文件系统
- 所有外部依赖必须可替换（mock/stub/fake）
- 不依赖真实时间、随机数或共享全局状态
- 避免顺序依赖与跨用例共享可变状态

## 覆盖要求

- 覆盖正常路径、边界条件、失败路径
- 重点覆盖分支判断与异常处理
- 对回归风险高的路径必须有用例

## 范围边界

- 只验证对外可见行为，避免测试内部实现细节
- 对异常路径需明确异常类型与关键信息

## 可读性与维护性

- Arrange 部分最小化，避免过度 setup
- 断言聚焦，不在一个用例里验证多个无关行为
- 重复 setup 使用 helper 或 fixture
- 复杂断言拆解为局部变量，提高可读性

## 常见反模式

- 用集成测试替代单元测试
- 一个用例覆盖多个场景
- 直接等待真实时间（sleep/timeout）
- mock 过深导致测试与实现强耦合

## 最小评审清单

- 是否明确场景 A（补测）或 B（测试先行）？
- 是否遵守结构与命名规范？
- 是否完全隔离外部依赖？
- 是否可重复运行且结果稳定？
- 是否覆盖边界与失败路径？
- 是否存在可读性问题？
- 是否避免依赖实现细节？
