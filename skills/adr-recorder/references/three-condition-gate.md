# 三条件 AND Gate 判断细则

> SKILL.md §2"必须材料"与 §3.1 Gate 步骤引用本文件。把"难逆转 ∧ 没上下文会惊讶 ∧ 真实取舍"具体化为可填、可机械校验的素材清单。

## 阈值由来(不是经验值)

- **三条件 AND**:任一条不满足就**不该写 ADR**——这是对"ADR 通胀"的反作用力。
  - 决定容易回滚 → 写代码注释或 PR description 即可,ADR 增加追溯成本但不增加追溯价值
  - 决定上下文已被项目文档充分覆盖 → 写在那篇文档里,ADR 重复冗余
  - 没有真实取舍(没认真考虑过别的) → 不是"决策",是"实施";不需要决策记录
- **机械校验**:三条件由 `check-adr-conditions.sh` 校验**字段非空且非占位**——这是对"用 ADR 走形式"的反作用力。LLM 自检容易给"性能更好"这类空话开绿灯,机械校验只能判结构性占位(`<...>` / `TODO`),实质内容是否空话由 review 阶段判。

两者结合:gate 不是要拦"每一份糟糕 ADR",而是拦"明显三条件之一缺失"的早期信号——后续的 review 与 6 个月后的回看才是真正校验"决策值得记"的回路。

## 三条件具体判断

### 1. 难逆转(irreversibility)

**填什么**:回滚成本量化。任一标尺即可:

| 标尺 | 阈值 | 例子 |
|---|---|---|
| 时间 | ≥ 1 周回滚工时 | 数据迁移回写、消费者切换 |
| 钱 | 直接付费(license / 云资源已 commit) | 已采购年付 SaaS |
| 对外契约破坏 | API 已发版,客户已集成 | gRPC proto 改名,SDK v2 已发 |
| 数据形态变更 | 表结构/事件 schema 已不可逆迁移 | 二进制日志格式变更后无法回滚 |
| 团队认知锁 | 技术栈切换后旧方向人员流失 | 主力团队已离职 |

**不算难逆转**:

- "性能更好" — 这是"决策的好处",不是"难逆转"
- "团队更熟悉" — 这是"决策的优势",不是"难逆转"
- "标准做法" — 这是"决策的合理性",不是"难逆转"

如果决策很容易回滚(改个配置、回滚一次部署就能回去),不该写 ADR。把这个决定记在 PR description 或 thinking-guardrails 输出里更合适。

### 2. 没上下文会惊讶(surprise-without-context)

**填什么**:列出 ≥ 1 条"6 个月后看到这个决定的合理疑问"。条件:

- 必须是"假设阅读者已经熟悉本项目其他部分"还会惊讶的点
- 必须是"如果你不留下解释,后人会做相反决定"的点
- 不能是"对全行业新人都会惊讶"——那不是上下文,是教科书

**好的例子**:

- "为什么订单服务用同步 RPC 而不是事件总线?——因为对账强一致性 SLA"
- "为什么不用 Postgres 的 JSONB?——历史上 schema 演进 5 次,JSONB 让 5 次都失控"

**反例**(不是"没上下文会惊讶"):

- "这是最佳实践" → 口号,不是上下文
- "团队选了这个" → 没回答 why
- "因为我们用 X" → 同义反复

如果一个新人看到决定**不会**有任何疑问,这条就不成立——决策对项目而言"显然",不需要 ADR。

### 3. 真实取舍(real-tradeoff)

**填什么**:被认真考虑过的备选 ≥ 2 个,各含**否决理由**(不只是"不合适")。

**阈值由来**:

- < 2 个备选:不算"取舍",算"实施"——你只是把唯一可行解写下来
- 2 个备选但都没否决理由:LLM 容易模式匹配填"备选 A 性能差,备选 B 复杂度高"——这种填法 review 会判失败,但 gate 不在结构层拦截(机械只能判非空)
- 2 个备选 + 各自否决理由:**形式过 gate,实质过 review**

**否决理由的"实质性"判断**(review 阶段,不是 gate):

| 形态 | 实质性 |
|---|---|
| "性能差" | 弱——多差?在什么 workload 下? |
| "在 100 QPS 下 p99 从 50ms 涨到 800ms,我们的 SLA 要求 p99 ≤ 100ms" | 强 |
| "运维成本高" | 弱——具体多少? |
| "需要新增 2 名 SRE 与 1 套监控栈,预算未批" | 强 |

**反模式**:倒推备选——决定已经做了,然后回去编 2 个不存在的备选凑数。这种 ADR 6 个月后回看会刺眼:备选明显是稻草人。

## Gate 与 Review 的分工

| 层 | 工具 | 拦什么 | 不拦什么 |
|---|---|---|---|
| Gate(本 skill) | `check-adr-conditions.sh` | 字段缺失 / 仅 `<...>` 占位 / 仅 `TODO` / 仅"反例:"指引行 | 空话(`"性能更好"` 是非空字符串,机械过 gate) |
| Review(人 / LLM 复读) | SKILL.md §6 验证清单 | 空话、稻草人备选、与项目其他文档冗余 | 远期回看价值——这要 6 个月后的人判 |

**关键**:Gate 不假装替代 Review。Gate 的价值是把"明显三条件缺失"的 ADR 在写之前就拦住,让 Review 集中精力判"实质内容"。

## 与"thinking-guardrails 奥卡姆剃刀"的边界

写代码前想要不要新增一个 wrapper / 配置项 / 抽象层 → 用 thinking-guardrails 的奥卡姆剃刀(去掉它会 break 什么),不是 ADR。

差别:

- 奥卡姆是**编码前置**:对单值/单参数/单 wrapper 的取舍,实时决定,不留快照
- ADR 是**架构层难逆转决策**:对模块/数据库/协议的取舍,事后留快照

如果你发现自己在为"加不加这个参数"写 ADR,基本是误用——用奥卡姆思考一下,大概率不需要这个参数,更不需要 ADR。

## 与"modeling-first 已知歧义"的边界(时间语义互斥)

modeling-first 的 `已知歧义` slot 是**活文档**:决策未定时记歧义,决策定了后**移除或标 `resolved by ADR-NNNN`**。ADR 是 **point-in-time 快照**:写下来之后**只能 supersede,不能改**。

不要做的事:

- 把 ADR 当 model 的副本,model 改了同步刷新 ADR——这违反 `Invariant.AdrRecorderSkill.2` 与 `Invariant.Process.cross.5`
- 在 ADR 正文加"如本决策被改动请同步本节"——ADR 不可改

正确做法:

- 决策被推翻 → 写新 ADR,旧 ADR 末尾追加 `> Superseded by ADR-MMMM`
- model 的"已知歧义"slot 由宿主 coding agent 收到 `AdrPublished` 事件后刷新(adr-recorder 不直接动 model 文件)
