# Feature: modeling-first skill 扩展

## Goal

将 modeling-first skill 从"后端领域建模专用"扩展为"coding agent 执行任何复杂任务前的通用理解框架"，覆盖逻辑代码、前端视觉与交互、复杂业务流程等场景，使 agent 通过建模理解任务而非依赖模式匹配。

## 来源

用户评审反馈 + 实际使用中发现的覆盖盲区（2026-04-15 对话）。

## Rules

### R1: 扩展建模词汇表

当前 skill 仅有"实体-关系-派生-不变量"四件套。必须新增以下建模维度，每种维度需定义：适用信号（什么时候用）、核心产出（输出什么）、与现有四件套的关系。

- **R1.1 状态机建模**：states / transitions / guards / actions。适用于：UI 组件有多状态、业务对象有生命周期、交互流程有步骤序列。作为独立建模维度，在 model.md 中以可选章节呈现。
- **R1.2 视觉领域派生**：视觉属性间的约束与派生（如响应式断点、间距系统、尺寸联动）。并入现有 Derivation Chains 章节，以"视觉领域"子类别呈现，不新建独立章节。在 modeling-guide 中增加视觉领域派生模式的说明。
- **R1.3 流程建模**：steps / conditions / branching / rollback / concurrency。适用于：多步审批、条件分支树、并发竞态、补偿/回滚逻辑。作为独立建模维度，在 model.md 中以可选章节呈现。
- **R1.4 组件识别建模**：跨页面/跨需求的可复用视觉单元识别。适用于：需求涉及多个页面/视图、设计稿中存在重复视觉模式。本质是 Reuse Check 在"实现前"阶段的前移——不是搜已有代码，而是在需求/设计层面识别共享单元。

### R2: 新增轻量建模模式

当前只有 outline（< 100 行文件）和 full（< 150 行文件）两种。必须新增：

- **R2.1 内联模式（Inline）**：不产出独立文件，在当前上下文中以 `## Inline Model` 标题输出 10-30 行结构化分析，随后直接进入实现。
- **R2.2 适用条件**：任务触发了建模信号（R4），但涉及单个组件/实体、≤ 2 个关系、无跨模块关注点。
- **R2.3 必须包含的最小结构**：至少覆盖"识别了什么"（实体/状态/组件）+ "约束是什么"（不变量/转换规则/派生关系）两个维度。免除锚点要求。
- **R2.4 可追溯性**：内联建模免除锚点和文件产出要求，但实现代码的结构应体现建模结论（如：建模识别了 3 个状态 → 实现中应有对应的状态定义）。

### R3: 优化建模深度判断逻辑

skill 保持原子性，激活机制不变（触发词 + 编排 skill 调用）。更新 skill 内部的判断逻辑：

- **R3.1 三级判断**：将"适用场景"从二分（建模/跳过）改为三级（跳过 / 内联 / 文件级）。
- **R3.2 判断信号表**：为每级给出明确的判断信号，覆盖后端、前端、流程三个领域。
- **R3.3 前端场景细化**：纯视觉调整（颜色、字号、圆角等不涉及状态或约束的改动）→ 跳过；涉及布局约束或状态变化的视觉改动 → 内联或更高。
- **R3.4 编排建议**：在 SKILL.md 中建议编排型 skill（如 spec-driven-dev）默认调用 modeling-first，而非等待用户主动触发。skill 本身不强制这一点。

### R4: 扩展建模触发信号

当前触发信号仅覆盖后端领域（新实体、新关系、新不变量、新派生、状态变化）。必须补充：

- **R4.1 前端触发信号**：组件有多状态（hover/active/disabled/loading...）、交互有步骤序列、布局属性间有联动、需求涉及多页面且存在视觉重复。
- **R4.2 流程触发信号**：多步操作、条件分支、并发/竞态、回滚/补偿。
- **R4.3 跳过条件细化**：保留现有跳过条件，新增"纯视觉调整（不涉及状态或约束）"为明确的跳过条件。

### R5: 组件识别建模的具体规则

- **R5.1 横向扫描**：在实现任何单个页面之前，先横向扫描所有涉及的页面/视图，提取重复的视觉单元。
- **R5.2 识别标准**（需求/设计层面，非实现层面）：视觉结构相同（设计稿/需求描述中布局一致）、交互模式相同（相同用户操作序列）、数据形状相同（接受相同类型的输入）——满足任意两项即判定为同一组件。
- **R5.3 产出**：共享组件清单（组件名 + 出现位置 + 输入接口草稿），放入 model.md 的可选章节或内联产出中。
- **R5.4 与 Reuse Check 的关系**：组件识别在 Reuse Check 之前执行。Reuse Check 检查"已有代码中是否有可复用的实现"；组件识别检查"当前需求中是否有可共享的视觉单元"。两者互补，不替代。

### R6: 非 CRUD 示例

- **R6.1**：modeling-guide.md 必须新增至少一个前端交互建模完整示例。
- **R6.2**：modeling-guide.md 必须新增至少一个复杂流程建模完整示例。
- **R6.3**：每个示例需覆盖：触发信号识别 → 选择建模深度 → 建模产出 → 如何指导实现。

### R7: 向后兼容

- **R7.1**：现有的"实体-关系-派生-不变量"四件套保持不变，新增维度是补充而非替代。
- **R7.2**：现有锚点系统保持不变，新增可选章节中的锚点使用相同格式（`<!-- anchor: StateMachine.<Entity>.<State> -->` 等）。
- **R7.3**：与 spec-driven-dev 的集成方式不变（仍作为 Step 0 硬依赖）。
- **R7.4**：现有 model.md / epic-model.md 模板的必填章节结构不变，新增内容以可选章节加入。
